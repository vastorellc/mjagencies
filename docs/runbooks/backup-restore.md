# Backup and Restore Runbook

**Audience:** On-call engineers and operators responsible for the MJAgency Postgres 17 instance.  
**Last updated:** 2026-04-25 (Plan 02-05)  
**Related:** `docs/runbooks/dr-drill.md`, `scripts/pgbackrest-restore.sh`, `scripts/pgbackrest-verify-r2.ts`

---

## Overview

pgBackRest is the canonical backup tool for the shared Postgres 17 instance (which hosts all 13 agency databases — brand, ecommerce, growth, webdev, ai, branding, strategy, finance, engineering, product, video, graphic, and the platform shared DB). pgBackRest provides:

- **Block-level incremental backups** — sends only changed 8 KB blocks after the initial full backup, making incrementals fast and small.
- **Native S3-compatible storage** — configured for Cloudflare R2 (`mjagency-backups` bucket), which has zero egress fees.
- **AES-256-CBC encryption at rest** — cipher passphrase managed in Doppler (`PGBACKREST_CIPHER_PASS`). Backup files in R2 are unreadable without the passphrase.
- **zstd compression** at level 3 — fast decompression (important for RTO), good compression ratio (~2–4x on Postgres data).
- **Continuous WAL archiving** — every WAL segment is pushed to R2 via `archive_command`, bounding data loss to the `archive_timeout` interval (60 seconds).

**SLA targets (PROJECT.md):** RPO 1h, RTO 4h.

**Backup config:** `infra/pgbackrest/pgbackrest.conf` (generated by `scripts/gen-pgbackrest-config.sh`)  
**Archive config:** `infra/pgbackrest/postgresql.archive.conf` (fragment appended to `postgresql.conf` at provisioning)

## Schedule

| Tier | Type | Schedule (UTC) | Retention | Notes |
|------|------|---------------|-----------|-------|
| WAL | Continuous archive-push | Every 60s (archive_timeout) | 1095 days (3 years) | Enables PITR to any point within retention |
| Daily | Incremental | 02:00 every day | 30 days (diff ladder) | Lightest I/O; ~5-50 MB/day with zstd |
| Weekly | Differential | 03:00 Sunday | 90 days (12 diffs) | Differential since last full |
| Monthly | Full | 04:00 1st of month | 84 months (7 years) | Aligns with REQ-018 vault retention |

Cron entries are in `infra/pgbackrest/cron/crontab`. Install as the `postgres` OS user:

```bash
crontab -u postgres infra/pgbackrest/cron/crontab
```

Alternatively, use systemd timers (preferred on Debian/Ubuntu — see `docs/runbooks/vps-provisioning.md`).

## Retention Bands

| Band | Retention | Purpose |
|------|-----------|---------|
| WAL archive | 3 years (1095 days) | PITR to any second within 3 years |
| Daily incrementals | 30 days | Fast recovery of recent data |
| Weekly differentials | 90 days (~12 sets) | Recovery within the quarter |
| Monthly fulls | 7 years (84 months) | Long-term compliance; aligns with REQ-018 vault |

The 7-year monthly full retention matches the Plan 02-06 permissions vault retention requirement (REQ-018). If vault retention changes, update `repo1-retention-full` in `infra/pgbackrest/pgbackrest.conf.tmpl` and re-run `scripts/gen-pgbackrest-config.sh`.

## Restore Procedure

### Restore Latest Backup

```bash
# 1. Stop Postgres
sudo systemctl stop postgresql

# 2. Restore (fetches latest backup + WAL from R2)
bash scripts/pgbackrest-restore.sh

# 3. Start Postgres (enters recovery, replays WAL, then promotes to read/write)
sudo systemctl start postgresql
```

Monitor recovery progress:
```bash
tail -f /var/log/postgresql/postgresql-17-main.log
```
Recovery is complete when you see `database system is ready to accept connections`.

### Point-in-Time Recovery (PITR)

Use PITR when you need to restore to a specific moment (e.g., just before an accidental data deletion):

```bash
# 1. Stop Postgres
sudo systemctl stop postgresql

# 2. Restore to a specific timestamp (ISO 8601)
bash scripts/pgbackrest-restore.sh --target='2026-04-26 14:30:00'

# 3. Start Postgres — it will replay WAL and stop at the target time
sudo systemctl start postgresql
```

**PITR granularity:** The minimum recovery granularity is determined by `archive_timeout = 60`. Any transaction committed within 60 seconds before the target time is guaranteed to be included. Data loss within a 60-second window is possible only if the Postgres host fails between WAL segment archives.

**Note:** The target timestamp is in the Postgres server's timezone (typically UTC on production VPS). Specify times in UTC to avoid ambiguity.

---

## PITR Limitations

- **Minimum granularity:** 60 seconds (bounded by `archive_timeout`). This satisfies the RPO 1h SLA with a comfortable margin.
- **WAL retention:** PITR is only possible within the WAL retention window (3 years / 1095 days). Older points require a full restore from a monthly backup with no WAL replay.
- **Cross-agency PITR:** All 13 agency databases are on the same Postgres instance and the same backup. PITR restores all databases simultaneously. Per-agency PITR is not supported at this architecture level.
- **Tablespace:** The default PostgreSQL tablespace is used. No additional tablespace configuration is needed for pgBackRest.

## Cipher Pass Rotation (SEC-10 Alignment)

pgBackRest does not natively support multi-version repo encryption. A cipher pass rotation requires a full re-key:

**Procedure:**

1. Generate a new cipher passphrase:
   ```bash
   openssl rand -base64 48
   ```

2. Update `PGBACKREST_CIPHER_PASS` in Doppler (shared project → Production environment).

3. Wait for the next scheduled monthly full backup to complete under the new passphrase, OR trigger a full backup immediately:
   ```bash
   sudo -u postgres pgbackrest --stanza=postgres-main --config=/etc/pgbackrest/pgbackrest.conf --type=full backup
   ```

4. After the new full backup (and its subsequent WAL) are confirmed in R2, old backups encrypted with the previous passphrase will expire naturally per the retention bands.

5. Verify with the smoke test (requires new key in environment):
   ```bash
   doppler run -- pnpm tsx scripts/pgbackrest-verify-r2.ts
   ```

**Important:** During the transition period (between new full backup and old backup expiry), operators must retain both the old and new passphrases. Store the old passphrase in Doppler under a versioned key (e.g., `PGBACKREST_CIPHER_PASS_PREV`) until all old backups expire.

## R2 Access Keys Rotation

When Cloudflare R2 access keys need to be rotated:

1. Generate new R2 API token in the Cloudflare dashboard (Account → R2 → API Tokens). Scope it to `Object Read & Write` on bucket `mjagency-backups` only.

2. Update `R2_ACCESS_KEY` and `R2_SECRET_KEY` in Doppler.

3. Re-generate pgBackRest config:
   ```bash
   doppler run -- bash scripts/gen-pgbackrest-config.sh
   ```
   This writes the updated `infra/pgbackrest/pgbackrest.conf`. Copy it to `/etc/pgbackrest/pgbackrest.conf` on the VPS.

4. Restart the cron service (or wait for next scheduled backup to auto-pick up the new config if it's deployed via the repo):
   ```bash
   sudo systemctl restart cron
   ```

5. Verify R2 connectivity:
   ```bash
   doppler run -- pnpm tsx scripts/pgbackrest-verify-r2.ts
   ```

**Note:** Old R2 API tokens remain valid until explicitly revoked in the Cloudflare dashboard. Only revoke after the new full backup completes successfully under the new keys.

## Smoke Testing

Run weekly in CI (configure as a scheduled job — target Plan 01-05 CI update):

```bash
doppler run -- pnpm tsx scripts/pgbackrest-verify-r2.ts
```

**What it tests:**
- Writes a probe object (`probes/pgbackrest-verify-<ts>.txt`) to the `mjagency-backups` R2 bucket.
- Reads it back and asserts the body is `ok` (verifies R2 write + read path).
- Lists objects under `archive/postgres-main/` (verifies WAL archive prefix is populated).
- Exits 0 on success, 1 on failure, 2 if `R2_ACCOUNT_ID` is unset (CI skip).

If the smoke test exits 1, check:
- R2 credentials in Doppler are valid and up-to-date.
- The `mjagency-backups` bucket exists in the Cloudflare R2 dashboard.
- Network connectivity from the VPS to `<account-id>.r2.cloudflarestorage.com`.

## Bandwidth

R2 egress is free (Cloudflare policy as of 2026). Backup upload costs:

| Operation | Estimated size | Frequency |
|-----------|--------------|-----------|
| WAL segment (16 MB) | ~1-3 MB compressed | Every 60s (when WAL fills) |
| Daily incremental | 5–50 MB | Daily |
| Weekly differential | 50–500 MB | Weekly |
| Monthly full | 500 MB – 5 GB | Monthly |

These are typical ranges for a 13-DB Postgres instance with moderate write volume. Actual sizes depend on data churn.

## Failure Diagnostics

**Symptom:** Backup cron job fails silently.  
**Check:** `journalctl -u cron | grep pgbackrest` and `/var/log/pgbackrest/` (pgBackRest own structured log). The cron piped to `logger -t pgbackrest-<tier>` so syslog entries show the exit code.

**Symptom:** R2 connectivity error ("Unable to connect to endpoint").  
**Most common cause:** R2 keys rotated in Doppler without re-running `gen-pgbackrest-config.sh` and redeploying `/etc/pgbackrest/pgbackrest.conf`.  
**Fix:** Re-run `doppler run -- bash scripts/gen-pgbackrest-config.sh`, copy to `/etc/pgbackrest/pgbackrest.conf`, verify with smoke test.

**Symptom:** "repo1-cipher-pass: cipher pass mismatch" during restore.  
**Cause:** `PGBACKREST_CIPHER_PASS` in runtime config does not match the passphrase used when the backup was created (cipher rotation in progress).  
**Fix:** Restore `PGBACKREST_CIPHER_PASS_PREV` to the old passphrase to restore old backups; use new passphrase for new backups.

**Symptom:** Restore completes but Postgres fails to start.  
**Check:** `/var/log/postgresql/postgresql-17-main.log`. Common causes: file permission mismatch (data dir must be owned by `postgres:postgres`), WAL gap (re-run restore without `--delta`).

**Symptom:** WAL archive falling behind (archive lag > 60s).  
**Check:** `SELECT pg_walfile_name(pg_current_wal_lsn());` and monitor `pg_stat_archiver.failed_count`. If failing, check the `archive_command` output in the Postgres log.
