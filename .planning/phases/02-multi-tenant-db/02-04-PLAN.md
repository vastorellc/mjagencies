<objective>
Implement the resumable per-agency seed framework defined in RESEARCH §3.1. The framework executes a list of `SeedStep` objects in declared order, tracks each step's status in the `_seed_state` table (created by Plan 02-01's schema), skips already-`completed` steps on re-run, marks `failed` on error with diagnostic `error_text`, and rethrows so a CLI-level retry resumes from the failed step. Every step's body runs inside `db.transaction()` with `set_config('app.agency_id', $slug, true)` set first (pitfall 8.5 — seeds without RLS context either fail visibility checks or silently insert wrong-tenant data).

Purpose: REQ-016 demands "seed scripts — transactional, resume on fail". This plan builds the executor + CLI + two real seed steps (the bare minimum needed to prove the framework). The two real steps — `agencies` (inserts the agency record) and `admin-users` (creates one super_admin per agency) — are the foundation Phase 3 (auth) needs to log in. Phase 5 (content sprint) and Phase 9 (CRM pre-seeds) plug in additional steps via the same `SeedStep` interface; nothing they ship requires changes to the executor in this plan.

drizzle-seed (RESEARCH §3.2) is installed but NOT used by this plan's two real steps — it's intended for bulk fake-data generation in test/dev. The framework supports drizzle-seed-style steps as a special case (just call `seed(db, schema).refine(...)` inside `step.run`), but the two seed steps shipped here are real-data inserts, not generated fakes. The `drizzle-seed` dep is added to `packages/db` so future steps (Phase 9 CRM pre-seeds, Phase 5 dev content stubs) can use it without re-evaluating the choice.

Plan-time decisions:
- **Steps shipped:** only `agencies` and `admin-users` at this plan. CRM pre-seeds (REQ-103, REQ-104) are Phase 9; content seeds (REQ-505) are Phase 5. The framework is the deliverable; the two real steps prove it works.
- **Per-agency vs shared:** RESEARCH §3.3 distinguishes per-agency steps (run × 12) vs shared platform steps (run × 1, against brand_db). At this plan, both shipped steps are per-agency — they run once per agency DB. Shared seeds are deferred to Phase 12 launch run.
- **`--all` parallelism:** runs the seed for all 12 agencies in parallel via `Promise.allSettled` (mirroring Plan 02-03's migration runner pattern). Within each agency, steps run sequentially — `_seed_state` ordering matters.
- **Reset path:** `--reset` truncates `_seed_state` BEFORE running. This is destructive — runbook clearly marks it dev-only.
- **Password handling for admin-users step:** the seeded super_admin gets a placeholder bcrypt hash for `'changeme'`. Phase 3 (auth) will replace this seed step's behavior to require a real onboarding token. Documented in step's JSDoc + runbook.

import type { AgencyDb } from '../client.js'

export interface SeedStep {
  /** Unique step identifier (becomes _seed_state.step_name primary key) */
  name: string
  /**
   * Step body. Runs inside withAgencyContext(...) — `tx` is already RLS-context-set.

}

Locked runner pattern (RESEARCH §3.1):
```ts
// packages/db/src/seed/runner.ts (skeleton)
import { eq } from 'drizzle-orm'
import { sql } from 'drizzle-orm'
import { seedState } from '../schema/seed-state.js'
import type { AgencyDb } from '../client.js'
import type { SeedStep } from './types.js'

export async function runSeed(
  db: AgencyDb,
  agencySlug: string,
  agencyId: string,   // UUID — must match agencies.id
  steps: SeedStep[]
): Promise<void> {
  for (const step of steps) {
    const existing = await db.select().from(seedState)
      .where(eq(seedState.stepName, step.name)).limit(1)

    if (existing[0]?.status === 'completed') {
      console.log(`[seed:${agencySlug}] ${step.name}: already completed, skipping`)
      continue
    }

    // Mark running
    await db.insert(seedState)
      .values({ stepName: step.name, status: 'running', startedAt: new Date() })
      .onConflictDoUpdate({
        target: seedState.stepName,
        set: { status: 'running', startedAt: new Date(), errorText: null },
      })

    try {
      await db.transaction(async (tx) => {
        await tx.execute(sql`SELECT set_config('app.agency_id', ${agencyId}, true)`)
        await step.run(tx, agencySlug)
      })
      await db.update(seedState)
        .set({ status: 'completed', completedAt: new Date() })
        .where(eq(seedState.stepName, step.name))
      console.log(`[seed:${agencySlug}] ${step.name}: completed`)
    } catch (err) {
      await db.update(seedState)
        .set({ status: 'failed', errorText: String(err) })
        .where(eq(seedState.stepName, step.name))
      throw err
    }
  }
}
```

// packages/db/src/seed/index.ts
import { createHash } from 'node:crypto'

/**
 * Derive a deterministic UUIDv5-style id for an agency slug.
 * Used by both the seed framework and the verify-pgbouncer-rls script.
 * The actual `agencies.id` UUID stored in the DB is derived from this function so
 * downstream tests/scripts can predict agency_id values without DB lookups.
 */
export function agencyUuid(slug: string): string {
  const h = createHash('sha256').update(`mjagency:agency:${slug}`).digest('hex')
  // Format as a v5-style UUID (deterministic; not a real UUIDv5 but fits PG uuid type)
  return [
    h.slice(0, 8),
    h.slice(8, 12),
    '5' + h.slice(13, 16),                  // version 5
    ((parseInt(h.slice(16, 18), 16) & 0x3f) | 0x80).toString(16) + h.slice(18, 20),  // variant 10xx
    h.slice(20, 32),
  ].join('-')
}
