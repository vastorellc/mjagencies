## Color

All values are `var(--mj-*)` references. Zero hex literals. OKLCH semantic tokens from `packages/ui/tokens/layer-2-semantic-color.css` and `layer-6-components.css`.

| Role | CSS Token | Usage |
|------|-----------|-------|
| Dominant (60%) | `var(--mj-color-bg-primary)` | Page background, calculator input area, Puck builder canvas, proposal page body |
| Secondary (30%) | `var(--mj-color-bg-secondary)` | Calculator result card background, proposal header band, invoice line-item zebra rows, Puck meta panel, admin bar background |
| Accent (10%) | `var(--mj-color-brand-500)` | See reserved list below |
| Destructive | `var(--mj-color-error)` | Invoice dispute badge, refund initiation button, proposal decline confirmation, chargeback panel header |
| Success | `var(--mj-color-success)` | Signed status badge, paid invoice badge, PDF ready confirmation checkmark |
| Warning | `var(--mj-color-warning)` | Benchmark expiry badge (REQ-406), proposal 14-day expiry countdown ≤3 days remaining |
| Info | `var(--mj-color-info)` | View tracking indicator (proposal viewed by recipient), invoice sent badge |

**Accent (`var(--mj-color-brand-500)`) is reserved exclusively for:**
1. Primary CTA buttons (one per surface: "Calculate Now", "Send Proposal", "Sign & Accept", "Pay Invoice", "Publish Page")
2. Active/focus ring on form inputs (`var(--mj-color-border-focus)` which resolves to `var(--mj-color-brand-500)`)
3. Puck admin bar "Publish" button
4. "Download PDF" and "Email My Report" primary action buttons

Accent is NOT used for: decorative borders, background fills, icon color in non-interactive states, status badges (those use semantic state tokens above).

> Per-agency themes override `--mj-color-brand-500` at `[data-agency="..."]` layer (Plan 04-04). All 12 OKLCH niche themes apply automatically — no component-level changes needed.

## Interaction Contracts

### Tool Calculator

- Primary visual anchor: tool result output number (Display, 36px, bold, `var(--mj-color-text-primary)`).
- Input fields: type="number" with min/max/step attributes matching benchmark dataset constraints. No LLM in calculation path (REQ-122).
- Calculate button: disabled until all required fields have valid values. Re-enables on any field change after first calculation.
- Result section: renders inline on same page below the input form — NOT a new URL route (REQ-413). `id="tool-result"` with `scroll-margin-top: var(--mj-space-16)` for smooth scroll after calculation.
- Benchmark expiry badge: rendered as a `<span role="status">` yellow pill using `var(--mj-color-warning)` background. Shown when benchmark age > 12 months (REQ-406). Tool remains fully functional — no lockout.
- PDF email gate: modal dialog (`role="dialog"`, `aria-modal="true"`). Opens only when "Get the Full PDF Report" is clicked. Email field autofocuses on open. Submitting calls server action → BullMQ → email send → CRM hook.
- Print/share: result section has `data-print-region="tool-result"` attribute. No separate shareable URL is generated (REQ-413 — inline only, not indexed).

### Proposal Builder (Recipient View)

- Signature pad: `react-signature-canvas` component. Canvas dimensions: full container width × 120px height minimum. Clear button always visible below canvas. Sign CTA disabled until canvas has strokes (non-empty).
- ESIGN disclosure: rendered as static text above signature pad. Must be visible without scrolling past it to reach the pad (vertical layout: disclosure → pad → Sign CTA).
- Status badges: `role="status"` element, updated via server-sent event or 30-second polling for view-tracking changes.
- Expiry countdown: rendered as `<time dateTime="...">` element. Switches from neutral to `var(--mj-color-warning)` background at ≤3 days remaining.
- Decline flow: two-step. First click opens an inline confirmation card (not a full modal). Requires explicit second click on "Yes, Decline".

### Invoice UI

- State machine transitions: draft → sent → viewed → paid/partial/disputed → refunded. States are non-reversible except partial → paid. Each state renders a distinct badge and a distinct primary CTA.
- Partial payment tracker: progress bar using `var(--mj-color-success)` fill on `var(--mj-color-bg-secondary)` track. Width = (amount_paid / total_amount) × 100%. Always shows both "Amount Paid" and "Remaining Balance" numerics.
- Chargeback evidence panel: read-only, collapsible (`<details>`/`<summary>`). Only visible when invoice status = `disputed`.
- Refund action: two-step confirmation. Button labeled "Initiate Refund" opens an inline confirmation card. Requires explicit "Confirm Refund" click. Destructive styling: `var(--mj-color-error)` background on confirm button.
- Pay Invoice button (recipient): links to Stripe Payment Link or PayPal checkout URL — opens in same tab (not new window) to preserve back-navigation.

### Puck Visual Builder (Admin)

- Admin bar: fixed position, top: 0, full viewport width, z-index above all page content. Height: 48px. Background: `var(--mj-color-bg-secondary)`. Contains: toggle label, SEO score widget, preview toggle, publish button.
- Builder access: server component wrapping `<Puck>` calls `requireSession()` and checks `session.agencyId === pageAgencyId` before rendering editor. Redirect to `/login` on failure. Cookie alone does NOT grant builder access (REQ-132, CLAUDE.md Puck Rules).
- Meta panel: right-side panel, 320px wide. Contains title, meta description, slug, SEO score widget. SEO score widget shows 0–100 integer score from `computeLiveScore()` with color ramp: 0–49 `var(--mj-color-error)`, 50–79 `var(--mj-color-warning)`, 80–100 `var(--mj-color-success)`.
- Preview mode: toggling preview hides the Puck edit chrome and renders the page as a visitor would see it. The admin bar remains visible with a "Back to Editor" button.
- Unsaved changes guard: `beforeunload` event listener active while editor has pending changes. Intercepts navigation and shows in-browser confirmation. Also fires on "Exit Editor" click — custom MJ dialog (not native `beforeunload` prompt) shown first.
- Publish: calls server action `publishPage(agencyId, pageId, puckData)` — begins with `requireSession()` auth check per CLAUDE.md rule 3. Success shows 3-second toast. Failure shows persistent error banner below admin bar.
- Block components in Puck: all sanitize string inputs before rendering (CLAUDE.md Puck Rules). No `dangerouslySetInnerHTML` from Puck JSON output.

## Accessibility Requirements

All Phase 10 surfaces must meet WCAG 2.2 AA (REQ-096, established in Phase 8):

- Calculator form: all inputs have `<label>` elements with `for`/`id` association. Error messages use `role="alert"` and are associated via `aria-describedby`.
- Email gate modal: `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing to modal heading. Focus trapped within modal while open. ESC closes modal.
- Signature pad: `<canvas>` has `role="img"` with `aria-label="Signature canvas. Sign using mouse, touch, or stylus."`. Clear button is a standard `<button>` element.
- Status badges: all use `role="status"` with `aria-live="polite"`.
- Invoice state machine: state transitions announce to screen readers via `aria-live="polite"` region.
- Puck admin bar: all controls are standard `<button>` or `<a>` elements. Not custom divs. `aria-pressed` on toggle.
- SEO score widget: score value includes accessible text label: `<span aria-label="SEO score: 72 out of 100">72</span>`.
- All touch targets: minimum 44×44px (WCAG 2.5.5).
- Color contrast: all text meets 4.5:1 on body, 3:1 on large text, using OKLCH tokens established in Phase 4.

## Security Notes (enforced at component layer)

- `<ToolResult>` — Phase 10 replaces the existing `dangerouslySetInnerHTML` stub with a sanitized renderer (DOMPurify on server side before writing to DB; client renders from safe stored HTML or JSON).
- Puck block components — all sanitize `string` props before rendering. No raw `dangerouslySetInnerHTML` from Puck JSON (CLAUDE.md Puck Rules).
- Email gate form — uses server action with `requireSession()`-free pattern (public form, same as Phase 9 `ContactForm` public route) but with HMAC-signed job payload before BullMQ enqueue.
- Signature canvas data — base64 PNG from `react-signature-canvas` is server-validated (non-empty, max 500KB) before PDF generation.
- Proposal public URL — token is a cryptographically random 32-byte hex string in the URL path. Not guessable. No auth required to view (public proposal), but POST (sign/decline) requires HMAC verification of the proposal token.
