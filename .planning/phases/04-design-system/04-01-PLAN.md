<objective>
Build the canonical 6-layer CSS variable token system for the MJAgency design system, plus the Tailwind v4 `@theme inline` bridge that exposes `--mj-*` tokens as utility classes. Every layer ships as a discrete CSS file under `packages/ui/tokens/` so Plans 04-03 (resolution stack), 04-04 (12 niche themes), and 04-05 (Storybook) can compose against a stable contract. The Phase 1 skeleton (`@theme { --color-brand-50: #f0f9ff; ... }`) is REPLACED — hex literals migrate to OKLCH primitives, and the bridge pattern becomes the single source of truth.

Purpose: Plans 04-02 (validator), 04-03 (resolution stack), 04-04 (12 niche themes), and 04-05 (Storybook visual regression) ALL consume this token surface. Without 04-01 there is no namespace, no schema, no Tailwind contract. The plan also resolves Open Q3 (Tailwind `@theme inline` smoke test) by building `web-main` and inspecting compiled CSS for `bg-brand-500` resolution.

Plan-time decisions captured (RESEARCH §1, §10):
- **Namespace prefix:** `--mj-` for all custom tokens. Tailwind's own `--color-*`/`--spacing-*`/`--font-*` defaults are NOT colliding because the bridge in `@theme inline` maps `--mj-color-brand-500` → `--color-brand-500` only at the utility layer. Components reference `var(--mj-*)` directly OR use Tailwind utilities (which resolve to the same vars at build).
- **Layer 1 = OKLCH primitives only.** Phase 1's hex literals (`#f0f9ff`, `#0ea5e9`, `#0c4a6e`) become `oklch(0.97 0.013 250)`, `oklch(0.55 0.19 250)`, `oklch(0.25 0.08 250)`. OKLCH is the standard for perceptual uniformity (verified Tailwind v4 docs, §1.2). All target browsers (modern Chromium, Firefox, Safari) support OKLCH (Assumption A1, §11).
- **Layer 6 isolation rule:** Component tokens (`--mj-btn-*`, `--mj-card-*`, `--mj-form-*`) NEVER reference primitives directly — they compose Layer 2-5. Plus illustration (`--mj-ill-*`) and icon (`--mj-icon-*`) sub-buckets in the same file. This guarantees that swapping the brand palette only requires touching Layer 1+2 — Layer 6 (components) inherits automatically.
- **Open Q3 resolution (`@theme inline` smoke test):** A verification step builds `apps/web-main` with `pnpm --filter=@mjagency/web-main build`, then greps `.next/static/css/*.css` for the resolved `bg-brand-500` utility. The expected output is a CSS rule referencing `var(--mj-color-brand-500)` (or the inlined OKLCH value if Tailwind inlines). If the smoke test fails, the bridge pattern needs adjustment.
- **Open Q3 resolution (`next/font/google` interaction):** `next/font/google` in Next.js 15 injects `--font-brand`, `--font-body` etc. as CSS vars on `<html>` at SSR time. Layer 3 (typography) defines `--mj-font-brand` with a CSS var indirection: `--mj-font-brand: var(--font-brand, 'Inter, system-ui, sans-serif')`. This means per-agency Next.js layouts can call `Inter({ variable: '--font-brand' })` and the token automatically picks up the agency-loaded font. Layer 3 documents this pattern in a comment.
- **Dark mode separation:** A separate `tokens/dark-mode.css` ships in this plan — but ONLY the Layer 2 color overrides flip. No asset reload, no class toggling on every component (REQ-046).

Output: `pnpm --filter=@mjagency/ui typecheck` exits 0. `pnpm --filter=@mjagency/ui vitest run` exits 0 with at least 8 token-shape tests passing. `pnpm --filter=@mjagency/web-main build` succeeds and `.next/static/css/*.css` contains the resolved `bg-brand-500` utility. The CI security-grep hex literal scan returns ZERO matches in `packages/ui/tokens/` and `packages/ui/styles/`.
</objective>
