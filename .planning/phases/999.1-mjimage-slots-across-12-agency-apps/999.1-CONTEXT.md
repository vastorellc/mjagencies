## Phase Boundary

**In scope:** Install `MjImage` (from `@mjagency/media`) call sites in 6 page templates across the agency apps that have those route files, with explicit `width`+`height`, non-optional `alt`, optional `blurHash` from the seed manifest, and `priority` only on hero/LCP slots.

**Out of scope (each must land first as its own phase):**
1. **Pre-A — Fix `@mjagency/media` exports.** `MjImage` is defined in `packages/media/src/picture.tsx` but is NOT re-exported from `index.ts`; `BlurHashResult` is referenced from `picture.tsx` but is NOT exported from `blurhash.ts`. Existing imports across the repo (e.g. `apps/web-ai/src/app/(frontend)/services/[slug]/page.tsx`) currently fail with `TS2307: Cannot find module '@mjagency/media'`. This must be repaired in a dedicated media-package phase before any new MjImage call site can compile.
2. **Pre-B — Image pipeline.** `agency-seed-manifest.ts` ships with 300 placeholder `cloudflare_image_id: ''` entries. The orchestration-layer image gate (commit `eb1f4e5`) rejects all real seed runs with `IMAGE SEED GATE FAILED — run image pipeline first` until those slots are populated with real Cloudflare Images IDs.
3. **Pre-C — Phantom-app scaffolding.** 9 of the 12 agency apps (web-construction, web-dental, web-financial, web-fitness, web-homeservices, web-legal, web-realestate, web-restaurant, web-spa) have orphan `(frontend)/tools/[slug]/page.tsx` files but **no `package.json`** — they aren't buildable Next.js apps. Even the 3 real apps (web-ai, web-branding, web-ecommerce) lack `@mjagency/media` in their dependency declarations. Scaffolding the 9 phantom shells + adding the media dep across all 12 must happen before slot-install.
4. **Pre-D — `case-studies/[slug]/page.tsx` route.** This route does not exist in ANY of the 12 apps. Scaffolding the route file (with `generateStaticParams`, `fetchPageBySlug` wiring, layout shell) belongs in the public-frontend extension phase, not in a slot-install phase.

**This phase delivers:** MjImage call sites, nothing else. No route scaffolding, no image production, no package surgery.

<decisions>
## Implementation Decisions

### Phase scope
- **D-01:** Scope is **slot installation only**. Phase 999.1 depends on 4 prerequisite phases (Pre-A through Pre-D above) — none of which exist yet on the roadmap. Promoting 999.1 to active sequence requires all 4 prerequisites to be planned and in flight or done. Researcher and planner MUST surface the prerequisite chain before producing a plan; do not generate a plan that silently inlines those prerequisites into 999.1.
- **D-02:** Final per-app reach depends on Pre-C: if all 12 apps are scaffolded, slots = 12 × 6 = 72; if only the 3 existing apps remain, slots = 3 × 6 = 18 (plus 5 tools/[slug] slots in apps that already have that route file). Plan must produce a coverage matrix asserting which apps × templates × slots are in scope at execution time.

### CMS source vs seed manifest
- **D-03:** **CMS is the runtime source of truth.** The seed manifest is the *initial* state that gets POST'd into Payload via `seed-payload-collections.ts`. After seeding, all page templates pull image data from CMS at request time via `fetchPageBySlug(AGENCY_ID, slug)` (existing pattern in `services/[slug]/page.tsx`). The seed manifest is NOT read by request-path code. This matches Phase 5 architecture and Phase 8 page-tree decisions.

### MjImage prop contract per template
- **D-04:** **`alt` is non-optional** at every call site. Empty-string `alt=""` is allowed ONLY for purely decorative images (none of the 6 template targets qualify; all are content images). Researcher must enumerate alt-text-source-of-truth per template (CMS field name, fallback rule).
- **D-05:** **`width` and `height` are explicit** at every call site, set to the source asset's intrinsic dimensions (CLS=0 contract, REQ-095). Defer the per-template dimension table to the researcher — it depends on the image-pipeline output spec from Pre-B. Common defaults to assume in researcher prompt: hero=1920×1080, secondary=1200×800, card=800×600, team=600×600, featured-image=1200×800.
- **D-06:** **`priority={true}` only on the hero slot per page.** Exactly one LCP element per page. All other slots use default `priority={false}`. Matches existing `services/[slug]` pattern.
- **D-07:** **`sizes` per template:** hero='(min-width: 1280px) 1200px, 100vw'; service-card / blog-card / team-tile = '(min-width: 768px) 50vw, 100vw'; featured-image (detail-style pages) = '(min-width: 1280px) 1200px, 100vw'. Researcher may revise based on grid breakpoints in `packages/ui` design tokens.
- **D-08:** **`blurHash` is wired from the CMS record** when present. Manifest already carries `blurhash` per record (interface added in commit `eb1f4e5`). The mapping from CMS → `BlurHashResult` shape is researcher's task (the `BlurHashResult` type itself must first be exported from `@mjagency/media` per Pre-A).
- **D-09:** **No raw `<img>` tags** anywhere in the 6 templates. Build gate: lint rule or codemod test that fails CI if a `<img` literal appears under `apps/*/src/app/(frontend)/`. Exact gate mechanism is a planner decision.

### Empty-state behavior
- **D-10:** **Build-time fail-loud:** `next build` MUST fail if any rendered page has a record with empty `cloudflare_image_id` (CLAUDE.md §5 content-complete). Implementation likely: a `generateStaticParams`-time assertion or a build-time check script over the seeded CMS state.
- **D-11:** **Runtime graceful degradation:** if a CMS edit later removes an image (post-launch admin action), the rendering page hides the affected card/tile rather than rendering `<MjImage cloudflareImageId="">` (which would 404 the asset URL). Detail-page heroes that lose their image fall back to the page's first non-hero image; if no images exist on the page, the entire page returns `notFound()`. Reasoning: don't ship dead-image holes in production but don't crash if a non-essential record loses its image.

### Test gates
- **D-12:** Each MjImage call site must be covered by an axe-core accessibility test asserting non-empty `alt` (existing axe-core gate from Phase 8 covers this if the templates render at test time).
- **D-13:** Per-app typecheck (`pnpm -F @mjagency/web-* typecheck`) must pass — currently broken on the 3 existing apps because of Pre-A.
- **D-14:** Phase 8's `runAxeTest()` pattern is the precedent for asserting alt text presence at component render time (see `apps/web-main/src/__tests__/...`). Researcher should propose extension to per-template snapshot.

### Claude's Discretion
- Per-template exact pixel dimensions (D-05) — researcher picks defaults from image-pipeline spec.
- Build-gate mechanism for "no raw `<img>`" rule (D-09) — planner picks lint rule vs codemod test.
- Whether the runtime fallback chain (D-11) lives in a shared helper in `@mjagency/media` or is inlined per template — planner decision.

</decisions>

- **`AGENCY_ID = '<slug>'` module-scope constant** at the top of every page route file. Phase 8 decision; new templates must follow.
- **`generateStaticParams` for `[slug]` routes** with `fetchAllPageSlugs(AGENCY_ID)` upstream call.
