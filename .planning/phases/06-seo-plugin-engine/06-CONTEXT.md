<decisions>
## Implementation Decisions

### Plugin parameter store (REQ-071, REQ-072)

- **D-01:** Plugin weights and rules live in the **existing `settings` Payload collection** — add a `seo_plugins` JSON field (not a new collection). One settings record per agency already exists from Phase 5. No new migration surface.
- **D-02:** Per-agency overrides use **merge-patch pattern**: global defaults defined in `packages/seo/src/plugin-defaults.ts`; agency `seo_plugins` field holds only the delta (overridden keys). Merge at read time: `{ ...pluginDefaults, ...agencyOverride }`. Resetting an agency to defaults = deleting the delta field.
- **D-03:** Plugin config is picked up **immediately on the next score compute** — `afterOperation` hook on `settings` collection save invalidates a Redis cache key (`agency:<id>:seo-config`). Score compute reads fresh config on cache miss. No background job delay.
- **D-04:** All 4 weight categories are admin-tunable per agency:
  - `seo_classic`: title length range, meta description length range, keyword density thresholds, internal link minimum, word count floor, H1/H2 presence scoring
  - `aio_citations`: required source types (government/academic/news), max citation age in months, whether unsourced stats block publish
  - `geo_chunking`: target radius in miles, city/region chunk count per page, whether geo variation is required for service pages
  - `score_thresholds`: minimum score to publish per plugin (e.g. seo-classic ≥70, aio-citations ≥60, geo-chunking ≥50)

### AIO TL;DR (REQ-075)

- **D-05:** TL;DR is **AI-generated by LiteLLM flash-lite on editor open** in the SeoPanel sidebar. Editor sees the draft and can edit it freely. Stored in a new `aio_tldr` field (string, max 120 chars) on the `pages` collection. `beforeOperation` publish hook blocks publish if `aio_tldr` is blank or exceeds 120 chars.
- **D-06:** TL;DR generation uses the existing `generateContent()` from `packages/ai` with a short summarize prompt. Falls back to empty string (not a blocker) when `LITELLM_API_URL` is absent (CI/local dev) — same pattern as Phase 5.

### FAQ schema (REQ-076)

- **D-07:** FAQs live in a **dedicated `faqs` Payload collection** (agency-scoped, same access rules). Pages collection gets a `faqs` **has-many relationship field**. Editor adds FAQ items when editing the page.
- **D-08:** The aio-citations plugin reads `page.faqs` on publish and auto-generates **FAQPage JSON-LD** injected into the page `<head>`. No editor friction beyond populating the FAQ relationship field.
- **D-09:** FAQ items are required to have `question` (string) and `answer` (richText or string) fields. `answer` field can be plain text for JSON-LD output.

### Algorithm watcher (REQ-074)

- **D-10:** Monitor **2 RSS feeds**: Google Search Central blog (`https://developers.google.com/search/blog/rss.xml`) + Search Engine Land / SEJ (configurable URL in Payload settings).
- **D-11:** Match trigger: **keyword list match** — configurable keyword array in Payload settings (instance-level, not per-agency). If RSS item title or description contains any keyword (e.g. `core update`, `helpful content`, `spam`, `ranking`, `algorithm`), it flags.
- **D-12:** On match: **creates a record in a new `algo_alerts` Payload collection** (global, not agency-scoped). Visible to `super_admin` in a dashboard panel. Alerts are dismissible/archivable. No email in Phase 6 (email engine is Phase 9 scope).
- **D-13:** Poll cadence: **BullMQ repeatable job every 6 hours** (`cron: '0 */6 * * *'`). Idempotent by RSS item GUID — already-seen GUIDs stored in Redis (`seo:algo-watcher:seen-guids` set, 90-day TTL).

### Claude's Discretion

- Score computation algorithm internals (how seo-classic computes the 0–100 score from weighted sub-scores)
- Redis key naming conventions for SEO config cache beyond the `agency:<id>:seo-config` prefix
- RSS parsing library choice (fast-xml-parser vs rss-parser — both acceptable)
- Exact plugin-defaults.ts data shape (as long as it satisfies the 4 tunable categories above)

</decisions>
