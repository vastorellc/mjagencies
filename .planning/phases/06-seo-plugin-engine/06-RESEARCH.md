## Summary

Phase 6 converts five Phase-5 stubs into a live, runtime-configurable SEO plugin engine. The core architectural challenge is threefold: (1) extracting structured data from Payload's Lexical richText JSON for scoring, (2) surfacing scores in real-time in the admin sidebar without blocking the editor, and (3) wiring two background jobs (algorithm watcher repeatable every 6h, self-learning loop periodic) into the existing BullMQ/instrumentation pattern.

All three plugins (seo-classic, aio-citations, geo-chunking) share a common scoring contract: a pure function that takes a `SeoPluginInput` and returns a sub-score 0-100 plus a findings array. The plugin engine aggregates these. Config is loaded from Redis (populated from Payload settings via afterOperation hook). The SeoPanel replaces its `computeSeoScore()` import with `useEffect` + debounce + server action call to the real engine.

The FAQ/JSON-LD and TL;DR features are the simplest deliverables: a new `faqs` collection, a relationship field on `pages`, and a server-side render of `<script type="application/ld+json">` in the page component.

**Primary recommendation:** Keep all scoring logic pure and synchronous in `packages/seo/src/plugins/`. The SeoPanel calls a Next.js server action with 500ms debounce; the server action loads config from Redis, runs all three plugins, and returns aggregate scores. This keeps scoring server-side (no bundle bloat, no browser-side parsing risk) and the panel stays a thin display component.

### Pattern 1: Lexical JSON Traversal

**What:** Walk `SerializedEditorState.root.children` recursively to extract plain text, headings, and links.
**When to use:** Any plugin that needs to analyze content from a richText Payload field.

```typescript
// Source: @payloadcms/richtext-lexical exported types + codebase pattern [VERIFIED: npm registry]
import type { SerializedEditorState, SerializedLexicalNode } from '@payloadcms/richtext-lexical'

export interface LexicalExtracts {
  plainText: string
  wordCount: number
  headings: Array<{ tag: 'h1' | 'h2' | 'h3' | 'h4' | 'h5' | 'h6'; text: string }>
  internalLinks: number
  paragraphs: string[]
}

function nodeText(node: SerializedLexicalNode): string {
  if ('text' in node && typeof node.text === 'string') return node.text
  if ('children' in node && Array.isArray(node.children)) {
    return (node.children as SerializedLexicalNode[]).map(nodeText).join('')
  }
  return ''
}

export function parseLexicalJson(raw: unknown): LexicalExtracts {
  const state = raw as SerializedEditorState | undefined
  if (!state?.root?.children) {
    return { plainText: '', wordCount: 0, headings: [], internalLinks: 0, paragraphs: [] }
  }

  const headings: LexicalExtracts['headings'] = []
  const paragraphs: string[] = []
  let internalLinks = 0

  function walk(nodes: SerializedLexicalNode[]): void {
    for (const node of nodes) {
      const t = node.type
      if (t === 'heading') {
        headings.push({ tag: (node as { tag: string }).tag as 'h1', text: nodeText(node) })
      } else if (t === 'paragraph') {
        paragraphs.push(nodeText(node))
      } else if (t === 'link' || t === 'autolink') {
        const url = (node as { url?: string }).url ?? ''
        if (url.startsWith('/') || url.includes(process.env['NEXT_PUBLIC_SITE_URL'] ?? 'INTERNAL')) {
          internalLinks++
        }
      }
      if ('children' in node && Array.isArray(node.children)) {
        walk(node.children as SerializedLexicalNode[])
      }
    }
  }

  walk(state.root.children as SerializedLexicalNode[])

  const plainText = paragraphs.join(' ')
  const wordCount = plainText.split(/\s+/).filter(Boolean).length
  return { plainText, wordCount, headings, internalLinks, paragraphs }
}
```

**Note on existing pattern:** `packages/cms/src/hooks/content-validators.ts` already uses `JSON.stringify(lexicalJson).replace(/"type":"[^"]+"/g, '')` for word counting. The full recursive walker above is more accurate for per-plugin analysis. Both can coexist.

### Pattern 2: seo-classic Scoring Algorithm

**What:** Weighted 0-100 score from 6 sub-factors. Weights are admin-configurable but use these defaults.
**When to use:** Score any indexable page content.

Industry-standard weight distribution based on Rank Math/Yoast analysis [CITED: rankmath.com/kb/score-100-in-tests]:

| Sub-factor | Default Weight | Notes |
|------------|---------------|-------|
| Title in target range (40-60 chars) | 20 pts | +partial if outside range |
| Meta description in range (120-160 chars) | 15 pts | Required on all pages |
| Focus keyword density (1-1.5%) | 20 pts | Warn + penalize >2.5% |
| Word count above floor (per page_type) | 25 pts | Graduated: 0→25 as count→floor |
| H1 present (exactly 1) | 10 pts | 0 if missing or >1 |
| H2 count ≥ 1 | 5 pts | Content structure signal |
| Internal links ≥ configured minimum | 5 pts | Already enforced by validator |

```typescript
// Source: derived from Rank Math scoring criteria [CITED: rankmath.com/kb/score-100-in-tests] + project conventions [ASSUMED weights]
export interface SeoClassicConfig {
  titleMinChars: number      // default: 40
  titleMaxChars: number      // default: 60
  metaDescMinChars: number   // default: 120
  metaDescMaxChars: number   // default: 160
  keywordDensityMin: number  // default: 0.01 (1%)
  keywordDensityMax: number  // default: 0.025 (2.5%)
  wordCountFloor: number     // default: 1500 (per page_type via pluginDefaults)
  internalLinkMin: number    // default: 3
}

export function scoreSeoClassic(
  extracts: LexicalExtracts,
  meta: { title?: string; metaDescription?: string; focusKeyword?: string },
  config: SeoClassicConfig
): number {
  let score = 0

  // Title (20 pts)
  const titleLen = meta.title?.length ?? 0
  if (titleLen >= config.titleMinChars && titleLen <= config.titleMaxChars) score += 20
  else if (titleLen > 0) score += 8 // partial credit

  // Meta description (15 pts)
  const metaLen = meta.metaDescription?.length ?? 0
  if (metaLen >= config.metaDescMinChars && metaLen <= config.metaDescMaxChars) score += 15
  else if (metaLen > 0) score += 5

  // Keyword density (20 pts)
  if (meta.focusKeyword && extracts.wordCount > 0) {
    const kw = meta.focusKeyword.toLowerCase()
    const kwCount = (extracts.plainText.toLowerCase().match(new RegExp(kw, 'g')) ?? []).length
    const density = kwCount / extracts.wordCount
    if (density >= config.keywordDensityMin && density <= config.keywordDensityMax) score += 20
    else if (density > 0 && density < config.keywordDensityMin) score += 10
    // >keywordDensityMax: 0 pts (keyword stuffing)
  }

  // Word count (25 pts — graduated)
  const wordPct = Math.min(extracts.wordCount / config.wordCountFloor, 1)
  score += Math.round(wordPct * 25)

  // H1 presence (10 pts)
  const h1Count = extracts.headings.filter(h => h.tag === 'h1').length
  if (h1Count === 1) score += 10
  else if (h1Count > 1) score += 0 // Multiple H1 is an SEO error

  // H2 presence (5 pts)
  if (extracts.headings.some(h => h.tag === 'h2')) score += 5

  // Internal links (5 pts)
  if (extracts.internalLinks >= config.internalLinkMin) score += 5
  else score += Math.round((extracts.internalLinks / config.internalLinkMin) * 5)

  return Math.min(100, score)
}
```

### Pattern 3: aio-citations Stat Detection

**What:** Detect sentences that assert a statistic but have no adjacent link node.
**When to use:** aio-citations plugin scoring.

```typescript
// Source: NLP stat detection patterns [ASSUMED — no authoritative library; regex-based approach]
// Stat indicators: percentage, dollar amount, "X times", "X out of Y", "according to"
const STAT_PATTERNS = [
  /\d+\.?\d*\s*%/,                          // percentage: "42%", "3.5%"
  /\$\s*\d[\d,]*/,                           // dollar amounts: "$1,200"
  /\b\d+\s*(?:times|x)\b/i,                 // "3 times", "5x"
  /\b(?:according\s+to|research\s+shows|studies?\s+(?:show|find|found|suggest))/i,
  /\b\d+\s*(?:out\s+of|in)\s*\d+\b/i,      // "3 out of 4", "1 in 5"
]

export interface CitationFinding {
  sentenceSnippet: string
  hasAdjacentLink: boolean
}

export function detectUnsourcedStats(
  paragraphs: string[],
  lexicalRaw: unknown        // raw Lexical JSON to cross-check link adjacency
): CitationFinding[] {
  const findings: CitationFinding[] = []
  for (const para of paragraphs) {
    const sentences = para.split(/[.!?]+/).filter(s => s.trim().length > 20)
    for (const sentence of sentences) {
      const hasStat = STAT_PATTERNS.some(p => p.test(sentence))
      if (!hasStat) continue
      // Approximate adjacency: check if the paragraph string contains a link URL near the stat
      const paraStr = JSON.stringify(lexicalRaw ?? '')
      const snippetIdx = paraStr.indexOf(sentence.trim().slice(0, 30))
      // Look for a URL within ~200 chars of the stat sentence in the raw JSON
      const window = snippetIdx > 0
        ? paraStr.slice(Math.max(0, snippetIdx - 100), snippetIdx + 300)
        : ''
      const hasAdjacentLink = /"url"\s*:\s*"https?:\/\//.test(window)
      findings.push({ sentenceSnippet: sentence.trim().slice(0, 100), hasAdjacentLink })
    }
  }
  return findings
}
```

**Scoring:** `aio_score = 100 - (unsourcedCount / totalStatCount) * 100` clamped to 0-100. If `unsourcedStats.blockPublish = true` in config, publish is blocked on any unsourced stat.

### Pattern 4: geo-chunking Scoring

**What:** Score geographic content density — count city/region entity mentions per chunk.
**When to use:** Service-area agencies (fitness, legal, medical, etc.).

Geo-chunking for local SEO in 2025 means: chunk content into ~800-token blocks, ensure each service-area page names the target city/region in: title, H1, at least 2 H2s, at least 3 body paragraphs [CITED: ibeamconsulting.com/blog/ai-seo-geo-content-chunking, searchengineland.com/geo-local-seo].

```typescript
// Source: GEO/local SEO content structure patterns [CITED: searchengineland.com/geo-local-seo-future-discovery] + [ASSUMED scoring thresholds]
export interface GeoChunkingConfig {
  targetRadius: number      // miles (informational, not computed here)
  targetCities: string[]    // city names to check — populated from agency settings
  chunkCountMin: number     // default: 2 chunks mentioning a city per page
  requiredOnServicePages: boolean  // default: true
}

export function scoreGeoChunking(
  extracts: LexicalExtracts,
  pageType: string,
  config: GeoChunkingConfig
): number {
  if (!config.requiredOnServicePages && pageType !== 'services') return 100 // not applicable

  let geoMentions = 0
  const fullText = [
    ...extracts.headings.map(h => h.text),
    ...extracts.paragraphs,
  ].join(' ').toLowerCase()

  for (const city of config.targetCities) {
    const pattern = new RegExp(`\\b${city.toLowerCase()}\\b`, 'g')
    const matches = fullText.match(pattern) ?? []
    geoMentions += matches.length
  }

  // Score: 0 mentions = 0, config.chunkCountMin mentions = 100
  const score = Math.min(100, Math.round((geoMentions / Math.max(config.chunkCountMin, 1)) * 100))
  return score
}
```

**Note:** `targetCities` must be populated in agency settings for meaningful scoring. When empty, geo-chunking score defaults to 0 with a "no cities configured" finding.

```typescript
// Source: Payload CMS docs — useAllFormFields hook [CITED: payloadcms.com/docs/admin/react-hooks]
// useAllFormFields returns [fields, dispatch] — re-renders on every field change
// useDocumentInfo gives collection + id for agency context

'use client'
import React, { useEffect, useRef, useState } from 'react'
import { useAllFormFields, useDocumentInfo } from '@payloadcms/ui'

// Server action defined in packages/cms/src/actions/seo-score.ts
import { computeLiveScore } from '@mjagency/cms/actions/seo-score'
import type { LiveSeoScore } from '@mjagency/cms'

export default function SeoPanel(): React.ReactElement {
  const [fields] = useAllFormFields()
  const { id, collectionConfig } = useDocumentInfo()
  const [scores, setScores] = useState<LiveSeoScore | null>(null)
  const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null)

  useEffect(() => {
    if (timerRef.current) clearTimeout(timerRef.current)
    timerRef.current = setTimeout(async () => {
      const agencyId = fields['agency_id']?.value as string | undefined
      if (!agencyId) return
      const result = await computeLiveScore({
        content: fields['content']?.value,
        metaTitle: fields['meta_title']?.value as string | undefined,

        focusKeyword: fields['focus_keyword']?.value as string | undefined,
        pageType: fields['page_type']?.value as string | undefined,
        agencyId,
      })
      setScores(result)
    }, 500) // 500ms debounce — avoids score on every keystroke
  }, [fields])
  // ... render 3 ScoreBars, TL;DR editor, Regenerate button
}
```

### Anti-Patterns to Avoid

- **Running scoring in the browser:** Lexical JSON parsing + keyword density regex over full content in the browser causes layout thrash and leaks scoring logic into the client bundle. Keep all plugin logic server-side.
- **Skipping the Redis config cache:** Reading Payload settings on every score request generates DB queries per keystroke. Always cache at `agency:<id>:seo-config` with a reasonable TTL (300s default; invalidate on settings save).
- **Global `algo_alerts` without super_admin access restriction:** The `algo_alerts` collection is global (no `agency_id`). Must use `superAdminOnly` access or it leaks algorithm intelligence to agency editors.
- **Storing Google service account JSON keys in code or git:** Use Doppler/env for `GSC_SERVICE_ACCOUNT_KEY` and `GA4_SERVICE_ACCOUNT_KEY`. Never commit key files.
- **Calling `queue.add({ repeat: { cron } })` inside a request handler:** Repeatable job registration runs at startup in instrumentation.node.ts. Calling it from an API route creates duplicate repeatable jobs on each request.
- **Trusting `JSON.stringify(lexicalJson)` for word counting alone:** The existing `countWords()` in content-validators.ts strips `"type":"..."` keys but still includes all JSON syntax characters. For per-plugin analysis, use the recursive walker in `lexical-parser.ts`.

## Common Pitfalls

### Pitfall 1: Lexical JSON shape is nested, not flat

**What goes wrong:** Treating Lexical JSON as plain text with regex — `JSON.stringify(node).replace(/"type":"..."/)` misses text nodes inside nested list items or table cells.
**Why it happens:** Lexical stores `SerializedEditorState.root.children` recursively. Headings contain children with the actual text nodes; paragraphs contain children with `SerializedTextNode`.
**How to avoid:** Always use recursive tree walking via `lexical-parser.ts`. Never assume text is at depth 1.
**Warning signs:** Word counts coming out much lower than visible text; H1 checks always failing even when H1 is visually present.

### Pitfall 2: useAllFormFields re-render storm without debounce

**What goes wrong:** Every keystroke in the Lexical editor triggers a re-render of SeoPanel, which fires a server action on each render, creating hundreds of concurrent requests.
**Why it happens:** `useAllFormFields` has no built-in debounce — it re-renders on every form field change. [CITED: payloadcms.com/docs/admin/react-hooks]
**How to avoid:** 500ms `useRef`-based debounce in `useEffect` before calling the server action (see Pattern 5).
**Warning signs:** Admin editor sluggish during typing; server logs showing rapid-fire score requests; Redis under high read load during editing sessions.

### Pitfall 3: afterOperation hook not firing on partial Payload updates

**What goes wrong:** When an editor saves only a single field (Payload autosave), the `afterOperation` hook on `settings` may fire with a partial `result` — `agency_id` may be absent if not in the update payload.
**Why it happens:** Payload's `afterOperation` receives the patched result, not the full document. `result.agency_id` could be `undefined` on partial saves.
**How to avoid:** In the `invalidateSeoConfigCache` hook, also accept the `id` from the hook args and do a lightweight Payload `findByID` to get `agency_id` if missing from `result`.
**Warning signs:** Cache not invalidating after settings changes; scores using stale weights.

### Pitfall 4: RSS GUID instability on Google Search Central feed

**What goes wrong:** Google Search Central RSS items sometimes have `guid` equal to the post URL rather than a stable unique ID. URL-as-GUID works fine until a post URL changes (e.g. canonicalization redirect), causing re-alert on an already-seen item.
**Why it happens:** RSS 2.0 spec allows `guid` to be any string; link and GUID can be identical.
**How to avoid:** Use `item.guid ?? item.link ?? item.isoDate + item.title` as the dedup key. Store full key, not just the last path segment.
**Warning signs:** Duplicate algo_alerts records appearing for the same post on consecutive runs.

### Pitfall 5: TL;DR generation called on every editor open

**What goes wrong:** Context.md D-05 says "generated on editor open" — if implemented naively, every editor open fires a LiteLLM call, even when `aio_tldr` is already populated.
**Why it happens:** Misreading D-05 — "on editor open" means "when the field is empty on open" (initial generation), not unconditionally.
**How to avoid:** In SeoPanel, only auto-trigger the "Regenerate" server action if `fields['aio_tldr']?.value` is empty or blank. Show a "Regenerate" button at all times for manual refresh.
**Warning signs:** LiteLLM usage spikes on each admin page open; existing TL;DR content getting overwritten.

### Pitfall 6: algo_alerts collection missing super_admin-only access

**What goes wrong:** `algo_alerts` is a global collection (no agency_id). If `collectionAccess` (which gates on agencyId) is used, no-one can read it. If `superAdminOnly` is NOT used, any authenticated user can read it.
**Why it happens:** New collection — easy to copy-paste wrong access pattern.
**How to avoid:** Use `superAdminOnly` for all access operations on `algo_alerts` and `seo_suggestions`. Only `super_admin` should see algorithm intelligence and AI tuning suggestions.
**Warning signs:** Agency admins able to view algo alerts in their Payload admin.

### Pitfall 7: Repeatable BullMQ job registered multiple times

**What goes wrong:** Calling `queue.add('check', {}, { repeat: { cron: '0 */6 * * *' } })` from instrumentation.node.ts creates a new repeatable entry each server restart, resulting in duplicate runs.
**Why it happens:** BullMQ adds a new repeatable key if the exact `{ name, cron }` combination doesn't match an existing entry. Slight differences (e.g. `data: {}` vs `data: null`) can cause duplicates.
**How to avoid:** Before `queue.add`, call `queue.getRepeatableJobs()` and skip add if a job with the same name and cron already exists. Or use BullMQ's `JobScheduler` API (v5+) which is idempotent by key. [CITED: docs.bullmq.io/guide/jobs/repeatable]
**Warning signs:** `seo:algo-watcher:seen-guids` Redis set growing faster than 6h intervals imply; duplicate algo_alerts records.

### rss-parser feed fetch with GUID dedup

```typescript
// Source: rss-parser 3.13.0 [VERIFIED: npm registry]
import Parser from 'rss-parser'
import type { Redis } from 'ioredis'

const SEEN_KEY = 'seo:algo-watcher:seen-guids'
const SEEN_TTL_SECONDS = 90 * 24 * 60 * 60 // 90 days

export async function processRssFeed(
  feedUrl: string,
  keywords: string[],
  redis: Redis,
): Promise<NewAlertItem[]> {
  const parser = new Parser()
  const feed = await parser.parseURL(feedUrl)
  const newAlerts: NewAlertItem[] = []

  for (const item of feed.items) {
    const guid = item.guid ?? item.link ?? `${item.isoDate}:${item.title}`
    if (!guid) continue

    const isSeen = await redis.sismember(SEEN_KEY, guid)
    if (isSeen) continue

    const text = `${item.title ?? ''} ${item.contentSnippet ?? ''}`.toLowerCase()
    const matched = keywords.filter(kw => text.includes(kw.toLowerCase()))
    if (matched.length === 0) continue

    newAlerts.push({ guid, title: item.title ?? '', link: item.link ?? '', matchedKeywords: matched })

    // Mark seen; pipeline SET + EXPIRE to reduce round trips
    const pipeline = redis.pipeline()
    pipeline.sadd(SEEN_KEY, guid)
    pipeline.expire(SEEN_KEY, SEEN_TTL_SECONDS)
    await pipeline.exec()
  }

  return newAlerts
}
```
